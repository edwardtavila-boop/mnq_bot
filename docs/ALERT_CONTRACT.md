# Pine Рєњ Executor Alert Contract

Pine strategies generated by `mnq.generators.pine` emit alerts whose
payload is a single-line JSON object. The executor parses these payloads
as the sole communication channel from TradingView to Tradovate.

This document is the **source of truth** for the JSON schema. Both the
Pine generator and the Python executor parser are built against it. Any
change here is a breaking change and requires bumping `schema_version`.

## Transport

- Pine emits exactly one JSON literal per `alert()` call.
- No trailing newline. No surrounding whitespace.
- `alert.freq = alert.freq_once_per_bar_close` for all alerts.
- Messages are always ASCII; non-ASCII identifiers are rejected at
  spec-load time.

## Common envelope

Every alert carries these fields:

| field            | type    | description                                                 |
|------------------|---------|-------------------------------------------------------------|
| `schema_version` | int     | Contract version. Currently `1`.                            |
| `event`          | string  | One of `entry`, `exit`, `cancel`, `diagnostic`.             |
| `spec_id`        | string  | `strategy.id` from the spec (e.g. `mnq_baseline_v0_1`).     |
| `spec_hash`      | string  | `strategy.content_hash` (e.g. `sha256:Рђд`) Рђћ lets the executor verify the Pine was generated from the exact same spec the executor holds. |
| `bar_time_iso`   | string  | ISO-8601 UTC timestamp of the bar close that produced the alert. |
| `bar_index`      | int     | TradingView `bar_index` at close. Used for dedupe.          |
| `symbol`         | string  | TradingView ticker (e.g. `CME_MINI:MNQM2026`).              |
| `nonce`          | string  | 16 hex chars derived from `str(time) + str(bar_index)` for idempotency. |

## Event payloads

### `entry`

Emitted when the Pine strategy's long or short entry condition fires on
a bar close. The executor is responsible for position sizing, bracket
construction, and venue order placement Рђћ Pine only says "intend to go
long/short now". No prices are sent; the executor uses live book data.

```json
{
  "schema_version": 1,
  "event": "entry",
  "spec_id": "mnq_baseline_v0_1",
  "spec_hash": "sha256:Рђд",
  "bar_time_iso": "2026-04-14T13:30:00Z",
  "bar_index": 12345,
  "symbol": "CME_MINI:MNQM2026",
  "nonce": "a1b2c3d4e5f60718",
  "side": "long",
  "stop_distance_ticks": 12,
  "take_profit_distance_ticks": 18,
  "context": {
    "ema_fast": 20250.25,
    "ema_slow": 20248.50,
    "vwap_session": 20245.00,
    "atr_14_ticks": 10,
    "rvol_20": 1.45,
    "htf_trend_rising": true,
    "session_window": "rth_open_drive"
  }
}
```

- `side` Рђћ `"long"` or `"short"`.
- `stop_distance_ticks` Рђћ integer ticks from entry to initial stop. Pine
  computes this from the spec's `exit.initial_stop` block and clamps to
  `min_ticks` / `max_ticks` locally as a cross-check. The executor re-
  clamps authoritatively.
- `take_profit_distance_ticks` Рђћ same shape for target.
- `context` Рђћ optional bag of diagnostic values at the bar close. Keys
  must match feature ids from the spec.

### `exit`

Emitted when Pine's local exit logic (its copy of stop/target/trail)
closes its local position. The executor treats this as *advisory*
(it has its own copy of stops running at the venue); the executor uses
this to detect divergence and raise an alarm, not to act.

```json
{
  "schema_version": 1,
  "event": "exit",
  "spec_id": "mnq_baseline_v0_1",
  "spec_hash": "sha256:Рђд",
  "bar_time_iso": "2026-04-14T13:42:00Z",
  "bar_index": 12357,
  "symbol": "CME_MINI:MNQM2026",
  "nonce": "Рђд",
  "reason": "take_profit",
  "r_multiple": 1.5
}
```

- `reason` Рђћ one of `initial_stop`, `take_profit`, `trailing`,
  `breakeven_stopped`, `time_stop`, `session_end`, `blackout`,
  `manual`.
- `r_multiple` Рђћ signed float, approx, for diagnostic only.

### `cancel`

Emitted when Pine withdraws a pending entry without a fill (e.g. the
cross condition unwound before the bar closed). The executor uses this
to cancel any matching pending limit orders it placed speculatively.

```json
{
  "schema_version": 1,
  "event": "cancel",
  "spec_id": "mnq_baseline_v0_1",
  "spec_hash": "sha256:Рђд",
  "bar_time_iso": "2026-04-14T13:31:00Z",
  "bar_index": 12346,
  "symbol": "CME_MINI:MNQM2026",
  "nonce": "Рђд",
  "intended_side": "long",
  "reason": "condition_failed"
}
```

### `diagnostic`

Optional heartbeat emitted once per N bars (N configurable; default off).
Lets the executor detect that Pine is still alive. Shape intentionally
open-ended Рђћ the executor logs and drops.

```json
{
  "schema_version": 1,
  "event": "diagnostic",
  "spec_id": "Рђд",
  "spec_hash": "Рђд",
  "bar_time_iso": "Рђд",
  "bar_index": 12400,
  "symbol": "Рђд",
  "nonce": "Рђд",
  "note": "heartbeat"
}
```

## Executor-side validation

On ingest the executor MUST:

1. Parse as strict JSON. Reject on any parse error.
2. Verify `schema_version == 1`.
3. Verify `spec_hash` equals the hash of the spec currently loaded by
   the executor. Mismatch РЄњ the Pine source is stale relative to the
   executor, refuse the alert, log an ERROR, and page.
4. Verify `nonce` has not been seen in the last 2048 alerts (rolling
   bloom). Duplicate РЄњ drop silently.
5. Verify `bar_index` is strictly greater than the last accepted
   `bar_index` for this `(spec_hash, symbol)`. Out-of-order РЄњ log
   WARNING and drop.

## Pine-side invariants

The Pine generator guarantees at build time:

- Every `alert(` call passes a JSON literal assembled via
  `str.format` / string concatenation of typed values. String fields
  are escaped through a `_json_str()` helper. No unescaped user input
  appears in the message body.
- `alert.freq_once_per_bar_close` is the only frequency used.
- Alerts are only emitted on confirmed bar close
  (`barstate.isconfirmed`), never on partial bars.
